## 2. Постановка задачи и контекст

1. **Предметная область и задача:**
   - проект решает задачу бинарной классификации физической формы пользователя;
   - потенциальный пользователь - человек, отслеживающий свою физическую форму.

2. **Формулировка задачи в терминах ML/ИИ:**
   - входные данные: 5 числовых признаков (возраст, рост, вес, продолжительность сна, индекс активности) и 2 категориальных (курение, пол);
   - целевая переменная - is_fit. Значения целевой переменной: 1 - в хорошей физической форме, 0 - не в форме;
   - сервис должен обеспечивать инференс ≤100 мс, ROC-AUC ≥0.80 и возможность интерпретации прогноза.

3. **Целевые метрики качества:**
   - метрики для оценки моделей: Accuracy (общая точность), Precision и Recall (детализация ошибок), F1-score (баланс точности и полноты) и ROC-AUC (основной критерий выбора модели);
   - выбор обусловлен умеренным дисбалансом классов (~60/40) и необходимостью оценивать не только итоговую классификацию, но и качество вероятностных предсказаний. Метрики вычисляются на отложенной тестовой выборке (20% данных) с сохранением распределения целевой переменной (стратификация). Финальная модель выбирается по максимальному ROC-AUC, при равенстве — по F1-score.

3. **Предобработка и EDA:**

   - важные наблюдения из разведочного анализа данных: умеренный дисбаланс классов (~60/40), пропуски в 4 признаках (до 15%), обработанные вставкой медианы/моды, сильная положительная корреляцию activity_index с таргетом (+0.343) и отрицательная — age (-0.214), реалистичные диапазоны признаков без критических выбросов.

основные шаги EDA и подготовки данных лежат в ноутбуке `notebooks/01_eda_baseline.ipynb`.

## 4. Модели и подходы

1. **Базовая (baseline) модель:**
   - в качестве baseline была использована логистическая регрессия;
   - результаты по ключевым метрикам: accuracy = 0.72, precision = 0.664, recall = 0.606, f1 = 0.634, roc_auc = 0.795.

2. **Улучшенная модель и эксперименты:**
   - поверх baseline была добавлена ансамблевая tree-based модель - ансамбль из 100 решающих деревьев (bagging), голосующих большинством;
   - max_depth - контроль сложности деревьев, защита от переобучения, stratify=y - сохранение распределения is_fit (60/40) в train и test;
   - проведены эксперименты: сравнение линейной и нелинейной архитектур (Logistic Regression vs Random Forest), отбор признаков на основе доступности для пользователя и статистической значимости, валидация регуляризации сложности деревьев через max_depth, тестирование устойчивости пайплайна предобработки, выбор метрики для принятия решения (ROC-AUC как устойчивая к дисбалансу).

Ноутбук: `notebooks/01_eda_baseline.ipynb`.

## 5. Экспериментальный протокол и результаты

1. **Экспериментальный протокол:**
   - выборка была разделена методом стратифицированного hold-out: 80% данных (1600 объектов) — на обучение, 20% (400 объектов) — на тест. Использовалась стратификация по целевой переменной is_fit для сохранения распределения классов (~60/40) в обеих подвыборках;
   - метрики качества рассчитывались на тестовой выборке. Ключевые параметры: zero_division=0 для устойчивости Precision/Recall/F1 к крайним случаям (когда модель не предсказывает положительный класс) и использование вероятности класса 1 (y_proba[:, 1]) для расчёта ROC-AUC.

2. **Сравнение моделей по метрикам:**


   | Модель / конфигурация         | Описание                             | ROC-AUC   | F1-score  | Комментарий              |
   |-------------------------------|--------------------------------------|-----------|-----------|--------------------------|
   | Baseline                      | LogisticRegression(max_iter=1000)    | 0.795     | 0.634     | Простая, интерпретируемая, быстрее обучается         |
   | Model A                       | RandomForestClassifier(n_estimators=100, max_depth=10)         | 0.776      | 0.643      | Не превзошла baseline по ключевой метрике ROC-AUC (0.776 < 0.795), несмотря на незначительный прирост значений других метрик|

3. **Выбор финальной модели:**
   - в качестве финальной модели выбрана LogisticRegression. 
   - значения метрик обеих моделей близки, но по ключевой метрике ROC-AUC выигрывает логистическая регрессия, что указывает на линейную зависимость в данных. Также выбранная модель имеет меньший вес, большую скорость и меньшую сложность;
   - вместо удаления строк с пропусками применена импутация медианой. Это позволило сохранить объём данных для обучения, что критично при N=2000;
   - исключены клинические метрики (пульс, давление), требующие оборудования, так как использование только self-reported признаков (возраст, вес, сон) снижает порог входа для пользователя, делая сервис массовым.

## 6. Архитектура решения и сервис

1. **Архитектура пайплайна:**
   Загрузка данных предоставляет сырой DataFrame, предобработка преобразует его в числовую матрицу через согласованный ColumnTransformer, модель обучается исключительно на этой матрице и сохраняет веса вместе с состоянием препроцессора. FastAPI-сервис не переобучает модель, а загружает сохранённые артефакты, применяет идентичный препроцессинг к валидированным входным данным и возвращает детерминированный прогноз. Такая связь гарантирует Training-Serving Consistency: параметры масштабирования и кодирования, вычисленные на обучающей выборке, идентично применяются на инференсе, что исключает искажение предсказаний и обеспечивает стабильную работу сервиса

2. **API и endpoints:**
   - endpoints реализованы: `/health`, `/predict`, `/features`;
   - эндпоинт POST /predict принимает JSON-объект с 7 полями: 5 числовых (age, height_cm, weight_kg, sleep_hours, activity_index) и 2 категориальных (smokes, gender). В ответ сервис возвращает три поля: предсказанный класс is_fit (0/1), вероятность probability (0.000–1.000) и текстовую рекомендацию message. При нарушении схемы ввода возвращается код ответа 422 с детализацией ошибки, при отсутствии загруженной модели — 503.

3. **Технологический стек:**
   - библиотеки и фреймворки: scikit-learn - обучение моделей; pandas - загрузка CSV, манипуляция DataFrame, фильтрация, агрегация; numpy - векторные вычисления; fastapi - REST-эндпоинты /predict, /health, /features автоматическая генерация Swagger UI; uvicorn - ASGI-сервер для запуска FastAPI; pydantic - валидация входных JSON; matplotlib и seaborn - построение графиков;
   - команды для поднятия сервиса:
```bash
cd project

uvicorn src.service.app:app --reload --host 0.0.0.0 --port 8000      
```

или

## 7. Наблюдаемость, конфигурация и безопасность

Кратко опишите:

1. **Логи и наблюдаемость:**
   - логирование сервиса построено по уровням: INFO фиксирует успешные предсказания, время инференса и загрузку артефактов; WARNING сигнализирует о превышении SLA (>100 мс) или аномалиях нагрузки; ERROR регистрирует ошибки валидации (422), недоступность модели (503) и критические исключения (500). 
   - корректность работы сервиса: эндпоинт /health возвращает model_loaded: true, стартовые логи подтверждают загрузку .pkl и состояния препроцессора, метрики инференса (латентность, доля ошибок) находятся в пределах SLA, синтетические запросы возвращают ожидаемые HTTP-коды.

2. **Конфигурации:**
   - параметры вынесены в конфигах:  `config.yaml`: target_column - имя целевой переменной, numeric_features - список числовых колонок, categorical_features - список категориальных колонок, гиперпараметры моделей, доля данных для тестов, `.env.example`: KAGGLE_USERNAME - логин для доступа к Kaggle API, KAGGLE_KEY - API-токен;
   - без изменения кода можно: провести эксперимент с новой моделью, изменив model.improved.type: xgboost, добавить новые признаки в numeric_features и categorical_features, сменить Kaggle-аккаунт.

3. **Безопасность:**
   - файл `.env` с секретами добавлен в `.gitignore` и не коммитится в репозиторий; в проекте присутствует только шаблон `.env.example`;
   - в репозитории нет чувствительных данных, в качестве примера присутствует только часть открытого датасета.
